# Architecture — Eastern-Astrology-MVP

> 东方命理（八字 + 紫微）× 西方心理占星（利兹·格林）双引擎。本文件为系统架构总览；产品需求见 Obsidian `P028-EasternAstrology/PRD.md`，调研依据见 `research/`。

## 1. 设计总原则

1. **计算层确定性、解读层柔性。** 星曜/宫位/四化/四柱/行星位置一律由开源库精确计算（"just math"，错一处即崩塌信任）；自然语言解读由 LLM 生成，措辞概率化、非决定论。**严禁 LLM 自行排盘**（杜绝幻觉）。
2. **统一命盘 Schema 是唯一接口。** 三引擎只填充 `UnifiedChart`；LLM 只读 `UnifiedChart`。
3. **东西双盘并置，谨慎共振。** 两套体系作为「同一个人的两面镜子」并行解读；只在 `RESONANCE_ANCHORS` 标注的主题交汇处做整合，禁止 1:1 等价（见 `research/liz-greene-...md §5`）。
4. **反思性、合规优先。** 心理占星 ≠ 临床心理；命理 = 自我反思工具而非预测神谕。强制免责声明 + 模型守护栏。

## 3. 数据流（核心解读路径）

```
出生信息(BirthInput)
  → [校验] Zod, <5ms
  → [归一] normalize: 真太阳时 + 子时归日 + DST，输出实际计算时刻(可审计)
  → [排盘] 三引擎并行 → UnifiedChart，<100ms（纯函数，按 BirthInput 缓存）
  → [事实抽取] 压缩为 load-bearing facts JSON（命宫主星+四化 / 日主+五行 / Sun-Moon-ASC+最紧相位）
  → [LLM] Claude Opus 4.8，冻结 system prompt 走 prompt-cache，streamed，结构化分段
  → [渲染] 命理 / 心理 / 建议 三段卡片 + 命盘图
```

## 5. 关键技术约定（陷阱）

- **iztro `timeIndex` 是 0–11(12) 时辰索引，不是小时**——用 `astro.getTimeIndex`，显式处理晚子时。
- **真太阳时 + 早晚子时**在 normalize 层统一处理，三引擎共用，保证四柱一致（影响日主）。
- **西方盘可降级**：缺纬度/时辰未知 → `western=null`，仅出东方双盘。
- **四化流派可配置**：iztro `configure({ mutagen })`，默认中州派；紫微解读层与流派绑定。
- **Swiss Ephemeris(AGPL) 规避**：用公有领域 circular-natal-horoscope-js（精度对占星足够）。
- **MVP 无需 DB / 无需账号**：命盘是 BirthInput 的纯函数，按需重算；缓存即可。
